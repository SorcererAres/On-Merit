# 竞品学习：JadeAI（AI 简历生成器）

> 对象：https://github.com/LingyiChen-AI/JadeAI （Apache-2.0，Next.js 16 全栈，自托管、免费、BYO-key）
> 目的：判断我们的 Resume Agent 从它身上**该学什么、别学什么**，以及由此得到的定位判断。
> 方法：clone 全仓（440 文件）通读文档 + 深读 AI 实现代码。文中 JadeAI 代码引用**以文件/函数为准，行号仅供定位、可能漂移**。
> 已过 GLM-5.1 跨模型复核（它联网核对了源码），据其修正，见文末〈复核记录〉。

---

## 〇 一句话结论

**JadeAI 用「功能广度 + 信任模型自由写」竞争；我们的差异化假设是「用 grounding/反编造做诚信」。它比我们成熟、可直接投产，且在「输出是否被真实材料约束」这条线上没有任何校验——这是我们选择的差异点。**

三点必须先说清，避免自我陶醉：
1. 「反编造是差异点」目前是**待验证假设**，不是已证的真理——无人打这张牌，可能是蓝海，也可能是没人要。需要需求验证。
2. **差异化需要载体**：反编造若没有可用的编辑/模板/预览体验承载，用户根本没机会感知到它。基础体验是前提，不是路线图末尾。
3. **判断「该不该学」的标准不是「编不编」，而是「对用户是否有真实价值、且局限能否被诚实标注」**——用同一把尺衡量它的每个功能。

## 一 它是什么（画像）

- **定位**：好用、完全免费、可自托管、AI 全用用户自己的 key、数据自持。
- **技术栈**：Next.js 16 · React 19 · TS · Tailwind4 + shadcn · dnd-kit · Zustand · Drizzle(PG+SQLite 双适配) · NextAuth v5 + FingerprintJS(指纹兜底免登录) · Vercel AI SDK(OpenAI/Anthropic/Gemini) · @react-pdf + Puppeteer · next-intl · Docker。
- **功能广度**（远超我们）：50 套模板 · 拖拽/行内编辑 · 撤销重做 · 自动保存 · AI 一键生成 · 图片/PDF 解析 · AI 对话优化 · 语法检查一键修复 · JD 匹配 · 求职信 · 10 语翻译 · **模拟面试(6 面试官+报告)** · 职业照生成 · 二维码 · 分享页+访问计数 · 多格式导出(PDF/DOCX/HTML/TXT/JSON)。
- **工程习惯**：语义化品牌色 token(`--brand-*`)、spec/plan 工作流、运行时环境变量——和我们同构。

## 二 核心洞察：分水岭是「输出是否被真实材料约束」

关键要**区分两类 AI 功能**，不能一刀切（这是上一版的错误，已改）：

**A 类 · 输入是用户真实材料 —— prompt 约束合理，风险是「准确性」不是「编造」**
- **简历解析（含图片 OCR）** `api/resume/parse`：输入是真实 PDF/图片，任务是 OCR + 抽取结构化。它的字段别名映射很包容。**缺的是「抽取准确性/完整性校验」（丢字、误读、日期错位），而不是反编造。** 我们的 `grounding_warnings` 正好补这一层。
- **语法/写作检查** `api/ai/grammar-check`（查弱动词/模糊/量化缺失，分 severity）、**基于已有内容的润色**——这些**不涉及编造，是合理且我们也应该做的**。反编造立场不等于拒绝一切 AI 改写。

**B 类 · 模型自由生成/拔高 —— 无 grounding = 可编造，这是它的真空白**
| 功能 | 实现 | 判断（已按复核收敛表述） |
|---|---|---|
| AI 一键生成简历 ⛔我们不做 | `api/ai/generate-resume`：给职位+年限+技能**可选 + 用户真实经历描述 `experience`** → 生成简历 | **可两用**：给了 `experience` 是结构化真材料，没给就靠模型编「可信公司名」。**问题是没有 grounding 去区分「结构化」与「编造」。我们只保留「结构化真实输入」，不做「凭空生成」。** |
| AI 对话优化（写回） | Vercel AI SDK + `updateSection` 工具改写并写回 | 基于已有简历改写（合理），但**无 grounding 拦软性拔高/凭空加数字**。 |
| JD 匹配 | `api/ai/jd-analysis`：纯 LLM 出 `overallScore`/`atsScore` + `keywordMatches`/`missingKeywords` | 关键词匹配/缺失**有信息论基础、有价值**；但 `atsScore` **不可复现、无确定性算法支撑**，不能当真实通过率。 |
| 模拟面试报告 ⛔我们不做 | LLM 逐题打分 + 能力雷达 | 纯主观、不可复现、不可作能力评级——与「不报不可复现分数」红线冲突。 |

**对比我们**：patch 模式（结构造假物理不可能）、逐字段 grounding（证据须出自简历）、刻意**不自评**覆盖度提升%、缺失只标「需真实补充」、每个 LLM 输出落地前有确定性校验。

→ **我们的护城河是「grounding + patch + 不自评」这条约束线，而不是「拒绝 AI」。** A 类功能（解析、语法检查、润色已有内容）我们完全可以做，而且做得更稳；B 类我们用约束替代放任——这才是差异点的准确表述。

## 三 值得吸收（按性价比排，均与我们栈兼容）

### A. BYO-key 自带密钥 —— 有价值，但别浪漫化
- **它怎么做**：key 存客户端 localStorage；请求时用 header（`x-provider`/`x-api-key`/`x-base-url`/`x-model`）注入；`src/lib/ai/provider.ts` 据此构造 provider。每个 provider 的配置独立保存、切回自动恢复、兼容旧名（`src/stores/settings-store.ts`）——这套配置缓存 UX 若做 BYO-key 必须一起抄。
- **准确表述（复核纠错）**：key **不被服务端持久化存储，但每次请求以 header 方式经过服务端内存**——不是「永不落服务端」。
- **它能解决**：数据出境/隐私、运营方 key 成本与简历留存。
- **它解决不了 / 的坑（复核补充）**：
  - **安全**：localStorage 里的 key 有 XSS 窃取面，需配套防护与提示。
  - **商业模式矛盾**：无服务端 key ⇒ 无法按量计费、限流、防滥用——适合开源自托管，和「可持续收费产品」天然冲突。
  - **不免真实性合规**：它只解隐私/出境，**不解「用户是否编造」**——那与 key 谁出无关。
  - **UX 门槛**：非技术用户不懂 API key；换浏览器/清缓存要重填。
- **结论**：**条件执行——面向开发者/自部署形态提供 BYO-key，不作为默认；面向普通用户仍需服务端托管 key（我们现在的 DeepSeek 方案）。优先级不排第一。**

### B. 分级 JSON 修复（自建，不假设它够强）
- **背景**：两次读代码对其修复强度判断不一——`api/resume/parse` 内的内联修复较简单（去围栏→取首末花括号→括号配平），是否另有更全的 `extract-json.ts`（去围栏→修引号→jsonrepair→暴力提取）存疑。
- **对我们的行动不受影响**：无论它多强，**我们都应自建一层分级修复**（去围栏 → 修未转义引号 → 括号/截断配平 → 兜底提取），因为我们现在只做「取首个 `{` 到末个 `}`」，LLM 偶发坏 JSON 会白重试。工作量小、收益直接。

### C. 灵活字段别名映射
- **它怎么做**：`name→fullName`、`job_title→jobTitle`、中英文字段名自动归一（`api/resume/parse` 的映射层）。
- **落地**：ingest 结构化后加一层归一化，减少「字段名不对→校验失败→重试」。工作量小。

### D. 多格式导出（DOCX 是刚需缺口）
- **它**：PDF/智能一页PDF/DOCX/HTML/TXT/JSON，Puppeteer 按模板服务端渲染。
- **我们**：只有浏览器 print-to-PDF；很多投递要求 Word。用 `python-docx` 从 JSON Resume 生成 DOCX；JSON 导入/导出便于迁移。工作量中。

### E. 求职信用自定义分隔符而非 JSON
- `TITLE:…\n---CONTENT---\n…`（`api/ai/cover-letter`）——长文本正文避开 JSON 转义。若我们加长文本生成直接用。

### F. 产品面（当路线图看）
版本快照（它自己都标为 P0 缺口，"最怕改坏回不去"）、分享页+访问分析、模拟面试（`InterviewerConfig` 角色卡 + interview-prompts 的「自然交互」提示词工程细致）、职业照、二维码、指纹免登录（零门槛开始用，自托管友好）。
> 凡涉主观 LLM 评分（面试报告等），一律套我们的「仅供参考、非招聘/面试率」标注——用统一标准，不双标。

## 四 我们的护城河（精确版）与红线

**护城河 = 约束线，不是「拒绝 AI」**：patch 模式（结构造假不可表达）+ 逐字段 grounding + 不自评 + 每个 LLM 输出前的确定性校验。这套是同类工具几乎都没有的。

**红线**：
1. 不做「无中生有造经历」——但「结构化用户提供的真实经历」「润色已有内容」「语法检查」都可做（属 A 类）。
2. 不报「ATS 通过率/面试率」这类不可复现的数字——用「证据覆盖指数（≠面试率）」替代，已落实。
3. 保留 patch + grounding + 不自评。

**明确不做（产品边界，已定）**：
- **AI 一键生成简历** —— 只给职位/年限/技能就生成整份简历，本质是替用户编造经历，直接违背反编造立场。用户的真实经历只能由用户提供（上传/粘贴/问卷），我们只做「结构化 + 不编造地改写」。
- **模拟面试报告** —— 纯主观 LLM 评分、无客观判分基准、不可复现，和我们「不报不可复现分数」的红线冲突。（模拟面试作为「练习/陪练」交互本身不排除，但**不产出被当成能力评级的报告**。）
> 这两项恰是 JadeAI 的 B 类核心功能；明确不做，等于把我们的差异点划得更清楚，而非能力缺失。

**我们相对它已有的优势（复核缺此上下文，补记）**：我们**已以 JSON Resume 为统一契约**（互操作性、生态验证器/渲染器免费获得），并**已有可用的三阶段编辑 + 自适应导出**。所以差距不是「编辑体验为零」，而是**模板丰富度与打磨**。

- 它 v0.3.4 独立做了和我们一样的**语义化品牌色 token**（`--brand-*` 干掉硬编码 `pink-*`）——印证我们 Geist token 化方向。

## 六 战略判断与落地优先级（已按复核重排）

**定位**：用「grounding/反编造」做差异化假设，但先保证差异化有可感知的载体；选择性吸收工程红利。

**落地顺序（BYO-key 已从第一降为条件执行）**：
1. **基础体验补齐**：模板丰富度 + 预览打磨（差异化的载体；没有它，反编造无人感知）。
2. **JSON 分级修复（§B）+ 字段别名映射（§C）**：小改动、直接增强现有链路鲁棒性。
3. **DOCX 导出（§D）**：补投递刚需缺口。
4. **A 类新功能**：语法/写作检查、基于原稿的润色（反编造立场下完全可做，且低风险）。
5. **BYO-key（§A，仅开发者/自部署形态）**：条件执行，非默认；普通用户仍走服务端托管。
6. （路线图）版本快照、分享页、职业照、二维码等——均带诚实标注。
   > 明确排除：AI 一键生成简历、模拟面试报告（见 §四〈明确不做〉）。

**待验证假设（不可跳过）**：如何验证「反编造」是有人愿意为之放弃部分功能丰富度的差异点？在验证前，把它当**方向假设**而非既定卖点。

**一句话**：JadeAI 教给我们的不是「该加什么功能」，而是「市面上的 AI 简历工具在 B 类（自由生成/拔高）上普遍无约束——这是我们选择切入的缝隙；但缝隙是否是市场，仍需验证，且必须有可用体验来承载」。

## 〈复核记录〉GLM-5.1 跨模型复核（已采纳）

由 `opencode -m siliconflow-cn/Pro/zai-org/GLM-5.1` 复核，它联网核对了 JadeAI 源码。采纳要点：
- **事实纠错**：BYO-key「永不落服务端」→ 改为「不持久化、但每次请求经过服务端」；generate-resume 可接收真实经历，非纯编造；ATS 分改为「不可复现、无确定性支撑」而非「编造」。
- **拆分立场先行**：区分 A 类（解析/语法检查/润色，输入真实、风险是准确性）与 B 类（自由生成/拔高，风险是编造）；承认 A 类我们也该做。
- **BYO-key 去浪漫化**：补 XSS/商业模式矛盾/不免真实性合规/UX 门槛，优先级降为条件执行。
- **框定为假设**：「反编造是差异点」改为待验证假设；判断「该学与否」统一为「有真实价值且局限可标注」，模拟面试与一键生成同尺衡量。
- **载体前置**：基础编辑/模板体验提到落地优先级首位。
- 未采纳/已澄清：JSON Resume 兼容——我们已是；「编辑体验为零」——我们已有三阶段编辑+导出（复核缺此上下文）；JSON 修复强度两读不一——结论收敛为「自建、不假设它够强」。
