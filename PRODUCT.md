# 产品方案：简历优化网站（修订版）

> 本文是把 `resume_agent/` 引擎产品化为面向求职者网站的方案，已吸收一轮跨模型（Codex）战略复核。
> 工程设计见 [DESIGN.md](DESIGN.md)，引擎用法见 [resume_agent/README.md](resume_agent/README.md)。

## 一、已定方向

- **市场**：国内为主（中文简历、国内岗位、PIPL 合规）
- **LLM**：托管 API 起步（通义千问 / 智谱 GLM，按量付费）
- **投入**：先做 MVP 验证，看真实反馈再加码

## 二、定位与差异化

**一句话**：把你的真实经历，针对**某个具体职位**讲到位——哪里够、哪里弱、哪里缺，一目了然，且每处改写都能在面试里解释。

**主卖点不是"AI 帮你改简历"（同质化），而是两点**：

1. **JD ↔ 简历证据映射**（核心差异）：不是给一个抽象的"78 分"，而是对**这份 JD** 逐条算覆盖度（已覆盖 / 证据弱 / 完全缺失）+ 证据。这回答用户真正的问题——"我这份对**这个岗**行不行"。抽象岗位评分 ≈ 直接用大模型，没有壁垒；JD 映射才有。
2. **诚信可辩护**（信任层，不做主标题）：改写只重述真事实，每处改动绑定原文证据、逐条可审；缺的（尤其硬性要求）标"需真实补充"，不替你编。讲法是"面试经得起追问 / HR 一眼信"，而不是道德说教。

> 反例（Codex 提醒）：把"诚信不造假"当主标题，用户会理解成"不帮包装、效果保守"而劝退。诚信是底线，不是购买理由。

## 三、引擎资产（护城河：智能内核已就绪）

产品化最难的"智能"部分已完成且经多轮跨模型评审、真机验证：

| 能力 | 模块 | 状态 |
|------|------|------|
| PDF → JSON Resume | `ingest.py` | ✅ + grounding 反幻觉核验 |
| **JD ↔ 证据映射** | `jd_match.py` | ✅ 覆盖度 + grounding 反造假（谎报匹配自动降级） |
| 按岗位评分 | `evaluate.py` + `rubrics.py` | ✅ 5 岗位 + 公平性脱敏 + 严格校验 |
| 事实约束改写 | `improver.py` / `patcher.py` | ✅ patch 模式结构造假物理不可能 |
| 针对 JD 改写 | `jd_match.improve_for_jd` | ✅ 强化弱项、缺失项标需补充 |
| 渲染精排 PDF | `kami_adapter.py` | ✅ Kami 编辑级模板 |
| 改动可审 | `resume_diff.py` | ✅ 逐字段 diff |

真机已验证：张良鹏真实 PDF → 自动结构化 → 对 AI 设计师 JD 覆盖度 94%→100% → 改进版 PDF，全程未编造数字、缺口正确指向"作品集链接"。**MVP 主要是给成熟引擎包壳。**

## 四、MVP = 邀请制付费试点（非公开上线）

> Codex 关键修正：原"4–6 周公开收费 MVP"验证的是"能否包成网页"，不是市场需求。改为**邀请制、人工兜底的付费试点**，真正验证意愿与效果。

**要验证的假设**：求职者愿不愿意为"针对具体岗位、诚信地把简历讲到位"付费，且真的拿去投递。

**核心流程（必须全有，不能砍）**：
1. 上传 PDF/DOCX/粘贴 → 结构化 → **解析预览 + 低置信字段确认纠错**（解析错一个公司/日期，整份就废，这不是高级功能）
2. **粘贴目标 JD** → JD↔证据映射：覆盖/弱/缺 + 硬缺口
3. 针对弱项**证据级改写**，**逐条接受/拒绝** + 导出前事实确认
4. 单模板 PDF 导出 + **真实支付**（单次 / 7–30 天求职包，先不做长期订阅）
5. 一键删除、明确保留期

**先不做**：多岗位对比、英文版、求职追踪、WYSIWYG 编辑器、多模板、雷达图（次要）。

**首发垂类**：选**文本密集型岗位**（产品 / 运营 / 数据），1–5 年经验、单栏简历。
> Codex 修正：不要选设计师——版式复杂难解析、判断高度依赖作品集（我们看不到）；"引擎评得好"≠"市场需求强"。

## 五、合规前置（W0 就做，不是 W5 文案）

国内上线的硬约束，时间线必须前置：

- **ICP 备案**：法定可达 ~20 工作日 → **W0 就提交**
- **生成式 AI 应用登记** + 公示上游模型名称/备案号（不默认模型厂商备案覆盖应用）
- **AI 生成内容标识**（2025.9 起已实施）：界面与导出 PDF 的显式/隐式标识方案要提前设计
- **PIPL**：简历=个人信息（照片/证件/政治面貌/健康等字段属敏感信息；含证明人等第三方信息）→ 上传前告知与单独同意、字段最小化、传输加密、删除闭环、**个人信息保护影响评估（PIPIA）并留档**
- 收费可能触及**经营性 ICP 许可**，按业务模式咨询接入商/属地主管
- 公安联网备案
- 第三方 LLM：选支持"不用于训练"的企业接口，并确认日志留存、人工审查、分包商、数据地域、删除能力

## 六、工程量诚实评估（别低估）

- **PDF 解析**：多栏阅读顺序、表格、页眉页脚、扫描件、旋转页、文字转路径、加密/损坏都会失败。首版**只支持单栏文本型 PDF/DOCX/粘贴**，扫描件走独立 OCR 路径，不假装 PyMuPDF 全覆盖。
- **中文渲染**：混排换行、字体回退、避头尾、长链接、跨页、溢出、Chromium 漂移、字体授权与嵌入。首版**一个单栏模板 + 内置可商用字体**，做"无截断/无重叠/字体已嵌入"自动回归 + headless Chrome 隔离（防注入/SSRF）。
- **成本**：LLM 单次调用便宜（每份几毛），但要算的是"每笔**成功付费导出**的全成本 + CAC"——含重试、解析失败、OCR、渲染、短信、存储、客服、退款、人工兜底。

## 七、技术栈（试点期从简）

- 前端 Next.js + Tailwind（ICP 备案）；后端 FastAPI（同语言直接包引擎）
- 任务：试点期用**托管队列或 DB 任务表**，先不上 Celery+Redis 全套
- 存储：PostgreSQL + 对象存储（PII 加密、签名 URL、默认 7 天自动删原文件）
- LLM：通义 qwen-plus / 智谱 GLM-4（`llm.py` 加一个 provider 即可）
- 渲染：容器内 headless Chromium + 内置 CJK 字体（不依赖 CDN）
- 登录：试点期手机号验证码 / 一次性链接即可（微信开放平台资质后置）

## 八、指标 & 决策阈值

**不看**虚荣指标（下载率被水印污染、改写接受率可灌水、低频产品留存先天弱）。

**核心指标**：完成事实确认并**实际投递**的**付费**简历数；7/14/30 天追投递、面试邀约、退款原因。

**继续投入阈值**（项目决策线，非行业基准）：
- ≥ 10 笔非人情付费
- 关键事实错误率 < 3%（人工复核金标集）
- 多数用户真的投了生成版
- 单一渠道 CAC 能被单笔毛利覆盖

## 九、冷启动

上线前定**单一渠道**（小红书内容 / 校招就业中心 / 垂直求职社群），人工招 **30–50 名非熟人**用户，关键输出人工复核。免费给诊断 + 局部预览，**付费解锁**完整改写 + 逐条采纳 + 无水印导出。

## 十、6 周节奏（邀请制试点）

| 周 | 交付 |
|----|------|
| W0 | 15–20 次目标用户访谈 + 定渠道/垂类 + **交备案** + 合规设计 + 50 份脱敏金标测试集 |
| W1 | JD 输入 + 单栏 PDF/DOCX/文本解析 + 低置信字段确认 |
| W2 | JD↔证据映射 + 证据级改写 + 逐条接受 + 真实模型质量评测 |
| W3 | 单模板 PDF + 支付 + 删除/退款 + 简易运营后台 |
| W4 | 邀请 30–50 名非熟人，关键输出人工复核 |
| W5–6 | 追真实投递与付费，决定 继续 / 换垂类 / 停止 |

> 公开收费上线另按 8–12 周估算；单人全栈更长。

## 十一、残留风险（诚实）

- **软性拔高**：patch 模式拦得住编造数字/结构，拦不住把真经历"往高里说"（真机已见：多端经验→"跨文化多市场能力"）。对策：逐条 diff 给人工审 + 文案提示"按真实情况增删"。
- **评分客观性**：无招聘结果校准前，覆盖度只是"对 JD 的证据匹配度"，**不**等于"提升面试率"——不要这样宣传。
- **PDF 丢字**：花哨简历解析不全 → 解析预览 + 粘贴兜底 + 字段确认。
